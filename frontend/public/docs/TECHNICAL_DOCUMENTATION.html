AgentForge Technical Documentation

v0.1.0 Python 3.12 FastAPI React 18 LiteLLM

1. Overview

AgentForge is a Linux desktop AI platform that combines a FastAPI backend, a React/TypeScript frontend, and LiteLLM for unified access to Ollama and cloud LLM providers. Agents can manipulate files and execute shell commands within a configured workspace, with human approval for non-whitelisted commands.

Core capabilities:

  • Single-agent coding mode with tool use (files, shell, memory)
  • Multi-agent orchestration with nine built-in roles and custom YAML roles
  • Quick, single, and multi orchestration modes
  • Mandatory command audit logging for all workspace and shell actions
  • Task-type based LLM routing (coding, SQL, research, documentation, …)
  • Persistent chat history in SQLite
  • Real-time streaming via WebSocket during agent execution
  • Setup wizard with connectivity and inference tests
  • Bilingual UI (English / German) via i18n JSON files

4. Request Data Flow

4.1 Chat message (REST)

  1. Frontend sends POST /api/chats/{id}/messages with content and mode
  2. AgentOrchestrator.run() loads chat history and memory context
  3. Model router selects LLM based on task type (if auto-routing enabled)
  4. LLM may return tool calls → tools execute → results fed back (up to 8 rounds)
  5. Shell commands outside whitelist create approval requests
  6. Final assistant message stored in SQLite; response returned to client

4.2 Real-time streaming (WebSocket)

  1. Frontend connects to WS /api/ws/chats/{id}
  2. User message sent as JSON (type: "message", content, mode, role_ids)
  3. AgentOrchestrator.run() detects workspace intent, prefetches/scans with command audit
  4. Server streams events: agent_start, tool_call, shell_command_recorded, agent_message, …
  5. On completion, a complete event with result is sent; on failure, error; on cancel, stopped
Full internal flow is documented in TECHNICAL_DOCUMENTATION.md (English, source of truth).

6. Workspace Intent Detection

Module: agentforge/agents/workspace_intent.py

Parses user text before LLM calls to detect disk actions:

  • wants_file_read — show file content in chat
  • wants_list_directory — list folder (takes priority over read/write keywords)
  • wants_file_creation / wants_directory_creation
  • wants_command_execution — shell

build_prompt_addon() injects strict tool instructions into the system prompt.

7. Task Board (Shared State)

Module: agentforge/agents/task_state.py

Blackboard shared across agents within one orchestration run:

  • TaskType: read_and_display, write_files, list_directory, run_command, general
  • TaskFact: verified info (reads, writes, listings) with source, path, agent, round
  • Persisted under memory key _agentforge_task_board between turns
  • Completion checks before final synthesis; weak retries (max 2) for incomplete answers

8. Command Audit (Mandatory Logging)

Module: agentforge/services/command_audit.py

Every workspace and shell action must be logged. Direct run_shell_command() calls are allowed only in command_audit.py and shell_security.py (enforced by tests).
FunctionSource
record_read_fileread_file tool, prefetch
record_list_directorylist_directory, workspace scanner
execute_approved_shell_commandApproved shell execution
record_commandGeneral shell audit entries

Uses command_audit_scope (ContextVar) so nested calls inherit chat/agent context. WebSocket events: shell_command_recorded, shell_command_pending.

9. Agent System

9.1 Orchestration modes

ModeEnumBehaviour
QuickquickFast answer without full tool loop
SinglesingleOne role with tool access (auto-resolve or selected)
MultimultiSelected roles discuss in rounds; PM synthesizes; task board

9.2 Built-in roles (9)

Defined in agentforge/agents/role_registry.py: developer, reviewer, architect, researcher, documentation, project_manager, software_tester, security, devops.

9.3 Custom roles

YAML files in assets/roles/ are loaded at startup:

id: security_expert
name: Security Expert
description: Audits code for vulnerabilities
system_prompt: You are a security specialist...
is_builtin: false

9.4 Tool execution loop

  • Maximum 8 tool rounds in single-agent mode
  • Maximum 4 discussion rounds in multi-agent mode
  • Unknown tool names trigger a bailout with user-visible error
  • JSON tool calls embedded in text content are parsed as fallback

10. Agent Tools

11.3 Model resolution

  1. If auto-routing is off → use default_model
  2. Detect task type from user content and active role
  3. Look up user registry assignment for that task type
  4. Fall back to assets/models.yaml priority list
  5. Verify model tag exists in Ollama via exact tag match

12. Memory System

Memory stores contextual facts the agent can recall across turns. Configurable per chat:

  • Token budget — how much memory context to inject
  • Scope — chat-only or global (shared across chats)

Memory entries are persisted in SQLite alongside chat data.

14. Security Model

14.1 Workspace sandbox

File tools resolve all paths under workspace_root. Escaping paths are blocked.

14.2 Shell command policy

PolicyDefault examplesAction
Whitelistgit, npm, python, ls, grepAuto-execute
Blacklistrm, sudo, curl, sshBlocked
UnlistedAny other commandUser approval required

14.3 API keys

Keys in backend/.env are local secrets. The settings API returns masked values. Keys entered via the UI are stored in the local SQLite database, not in the repository.

14.4 Network

Backend binds to 127.0.0.1 by default — not exposed to the network unless configured otherwise.

16. Launcher (run.py)

The launcher script orchestrates all processes:

  1. Optionally prompts for desktop shortcut setup
  2. Starts backend if not healthy on port 8765
  3. Starts Vite dev server if not healthy on port 5173
  4. Opens UI based on AGENTFORGE_MODE env var or auto-detection
ModeEnv valueResult
Auto (default)—Tauri binary → Tauri dev → browser window → browser tab
Browser tabbrowserxdg-open URL
App windowwindowChromium/Firefox --app mode
Tauritaurinpm run tauri:dev

Stop with python3 stop.py — kills PIDs and frees ports 8765 and 5173.
